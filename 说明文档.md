# AgentMonitor 说明文档

## 目标
- 使用 `demo-agent/` 通过本仓库 `sdk/` 向平台后端上报数据，用于验证“平台入库与查询展示的数据是否正确”。

## 范围
- 仅覆盖 demo-agent → SDK → 后端接收/入库 → 前端/接口查询 的基础数据链路验证。
- 不扩展断点规则拉取、快照上报等高级能力的完整闭环（若需要另行补充规划）。

## 现状梳理（关键结论）
- `demo-agent` 从环境变量读取 `MONITOR_API_KEY` 与 `MONITOR_API_URL`，实例化 `sdk` 并在对话/工具调用中产生 trace 上报。
- `sdk` 默认向 `{baseUrl}/api/v1/traces` 发送 `POST`，并使用 `Authorization: Bearer <apiKey>` 作为鉴权头。
- 后端用于“Agent 上报”的鉴权中间件要求 `X-API-Key` 请求头；同时后端在 `NODE_ENV=test` 时 API 前缀为 `/api`，否则为 `/api/v1`。
- 若不处理上述差异，demo-agent 可能出现 401（缺少 X-API-Key）或 404（API 前缀不一致），从而导致平台侧看不到 demo 数据。

## 方案
- 调整 SDK 上报请求，兼容后端鉴权与 API 前缀：
  - 上报类请求同时携带 `X-API-Key`（与后端一致）。
  - API 前缀支持配置，并在默认 `/api/v1` 失败时回退 `/api`（适配测试环境）。
- 运行 demo-agent 产生数据，并通过后端查询接口/前端页面进行对照验证。
- 对齐 `/stats` 接口返回字段与前端仪表盘卡片字段，避免统计展示为 0。
- 修复前端开发环境 WebSocket 连接：
  - Vite 增加 `/ws` 代理到后端，避免页面一直显示“重连中”。
  - 前端 WS 连接后发送 `subscribe` 消息订阅当前 project。
- 实现总 Token 数统计：
  - 后端从 `traces.metadata.usage` 等字段汇总 tokens。
  - demo-agent 增加 mock LLM 调用并上报 usage，用于验证 tokens 链路。
- 修复成本分析页面数据链路：
  - 后端成本统计按 SQLite/PostgreSQL 分别实现兼容 SQL（避免 SQLite 下使用 `::text` 等方言导致 500）。
  - 前端成本页接口请求使用 `Promise.allSettled`，避免单接口失败导致整页空白/全为 0。
  - 模型价格匹配按“更具体优先”（例如 `gpt-4o-mini` 不应被 `gpt-4` 规则抢先命中）。
- 优化断点触发历史展示：
  - 后端查询快照时将 `state` 从 JSON 字符串反序列化为对象，避免前端二次 stringify 导致转义长串。
  - 前端触发历史的 JSON 详情增加自动换行与最大高度，提升可读性。
- 告警功能使用与限制：
  - 前端可配置告警规则（延迟/错误率/成本/自定义）并查看告警历史。
  - 当前告警规则与历史存储在后端内存中，服务重启会丢失；且默认不会自动检测，需要手动触发检测（前端提供“立即检测”按钮，对应 `/api/v1/alerts/check`）。
  - 告警历史会记录触发时的关键排查信息：告警类型、阈值、触发实际值、指标快照（平均延迟/错误率/近 1 天成本）与自定义表达式（如有）。
  - 告警为“项目级指标”触发，无法严格归因到某一次调用；历史中提供“关联调用/会话样本”（最近 24 小时内最慢/最近错误/最贵的一条 trace）用于快速定位排查。
  - 支持“忽略同类告警（基于告警历史）”：对某条告警历史执行忽略后，会在指定时长内抑制同一规则产生的重复告警内容（下一次点击“立即检测”不会再新增同类告警历史）；到期后自动恢复。

## 风险与关注点
- 后端服务当前运行环境的 `NODE_ENV` 会影响 API 前缀；需要确保 demo-agent 的 SDK 选择正确前缀。
- 平台“断点/快照”等高级能力存在接口/鉴权差异，可能需要单独修复后才能完整演示。
- 成本统计依赖 traces 的 token usage 字段：
  - 若只上报了 session/message/function 等非 LLM trace，或 metadata 未携带 tokens，成本可能为 0。
  - 后端已兼容 `prompt_tokens/completion_tokens` 与 `input_tokens/output_tokens` 等常见字段命名，但仍需要上游实际传入 usage。
- 延迟展示口径：
  - 列表中 `latency_ms` 为 `null/undefined` 会显示 `-`，表示该类事件未记录耗时（例如 message/session）。
  - 平均延迟来自后端 `AVG(latency_ms)`（仅统计有 latency 的记录），前端展示统一保留 1 位小数。
- 质量分析口径：
  - 质量分数=速度分数(60%)+成功分数(40%)，速度分数由 `latency_ms` 分段计算，成功分数由 `status` 是否为 `success` 决定。
  - 质量趋势在 SQLite/PostgreSQL 下分别使用对应 SQL 聚合；前端即使趋势接口失败也会优先展示总分（避免全为0）。
  - Trace 写入时若未提供 `status`，后端会按是否存在 `error` 自动推断为 `success/error`，避免默认 `pending` 导致质量分析“基于0条记录”。
